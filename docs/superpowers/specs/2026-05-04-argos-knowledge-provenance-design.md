# Argos Knowledge Provenance Design

Date: 2026-05-04

## Goal

Add a Git-native provenance layer for knowledge publishing. Argos should record
machine-checkable evidence for each knowledge write and publish attempt without
pretending to replace human or team review.

The target model is:

```text
design = what knowledge should be written and why
draft = inbox knowledge content
provenance = how this draft was designed, checked, decided, and published
PR review = team review of the knowledge diff and provenance evidence
main branch = team accepted knowledge boundary
```

`provenance` is intentionally chosen over `review`, `approval`, or `release`:

- It matches established supply-chain vocabulary such as SLSA, in-toto, Sigstore,
  and GitHub artifact attestations.
- It describes origin and process evidence without claiming that Argos itself
  performed real human review.
- It fits both personal local use and team PR-based use.

## Source Of Truth

Argos first-release storage remains Git-native:

- Git-tracked Markdown/YAML/JSON files are the canonical knowledge store.
- SQLite indexes are generated caches and not source of truth.
- Object storage such as MinIO or S3 is a later optional attachment store for
  large assets, not part of the first-release trusted publish chain.

Trust depends on the Git ref:

```text
feature branch official path = official candidate for review
main branch official path = team accepted official knowledge
```

This distinction is important. `argos knowledge publish` may move a draft to an
official path on a feature branch, but the team has not accepted that knowledge
until the branch is merged through the repository's normal controls.

## Non-Goals

This module does not add:

- GitHub, GitLab, or other PR platform API dependencies.
- A permissions system.
- Cryptographic signing or certificate verification.
- A remote metadata service.
- A full append-only event database.
- A human-facing review UI.

Those can be layered on later through external review metadata, signed
attestations, or protected branch policy.

## Storage Layout

Each write attempt has one provenance ID.

While the draft is in inbox:

```text
knowledge/.inbox/provenance/<provenance-id>/
  provenance.json
  decisions.jsonl
  checks/
    check-001.json
```

After publish:

```text
knowledge/provenance/<knowledge-id>/<provenance-id>/
  provenance.json
  decisions.jsonl
  checks/
    check-001.json
```

The provenance directory is Git-tracked. It should be reviewed in the same PR as
the knowledge content it supports.

## Provenance ID

`provenance-id` should be generated by Argos, not hand-written.

Recommended shape:

```text
prov-<date>-<slug>-<short-random>
```

Example:

```text
prov-20260504-redis-cache-a1b2c3d4
```

The ID needs to be stable after creation and unique within the repository. It
does not need to encode the final knowledge ID because the final ID may not be
settled when the provenance record starts.

## `provenance.json`

`provenance.json` is the summary record and current binding for the publish
attempt.

Draft example:

```json
{
  "schema_version": "knowledge.provenance.v1",
  "provenance_id": "prov-20260504-redis-cache-a1b2c3d4",
  "state": "draft",
  "subject": {
    "kind": "package",
    "knowledge_id": "package:mall-api.redis-cache.v1",
    "project": "mall-api",
    "design_path": "knowledge/.inbox/designs/mall-api/redis-cache/design.json",
    "draft_path": "knowledge/.inbox/packages/mall-api/redis-cache",
    "official_path": "knowledge/packages/mall-api/redis-cache"
  },
  "hashes": {
    "design_sha256": "hex...",
    "draft_tree_sha256": "hex...",
    "latest_check_sha256": "hex..."
  },
  "latest_check": {
    "path": "checks/check-001.json",
    "result": "pass"
  },
  "created_at": "2026-05-04T00:00:00Z",
  "created_by": "codex"
}
```

Published example additions:

```json
{
  "state": "published",
  "published_at": "2026-05-04T00:10:00Z",
  "published_by": "codex",
  "published_from": "knowledge/.inbox/packages/mall-api/redis-cache",
  "published_to": "knowledge/packages/mall-api/redis-cache"
}
```

`created_by`, `published_by`, and decision `recorded_by` identify the actor that
recorded or executed the Argos operation. They do not prove human authority.
Team authority comes from Git history, PR review, CI, CODEOWNERS, and protected
branches.

## `decisions.jsonl`

`decisions.jsonl` stores one JSON decision per line. These are decision records,
not platform-enforced approvals.

Example:

```json
{
  "schema_version": "knowledge.provenance.decision.v1",
  "decision_id": "dec-001",
  "stage": "design",
  "decision": "approved",
  "decided_by": "chenchi",
  "decider_role": "knowledge_owner",
  "recorded_by": "codex",
  "source": "conversation",
  "reason": "The Redis cache package scope and boundaries are correct.",
  "hashes": {
    "design_sha256": "hex..."
  },
  "created_at": "2026-05-04T00:02:00Z"
}
```

Supported first-release stages:

- `design`
- `draft_write`
- `publish`

Supported first-release decisions:

- `approved`
- `changes_requested`
- `rejected`

Supported first-release sources:

- `conversation`
- `pull_request`
- `issue`
- `document`
- `other`

`publish` decisions must bind to the current design hash, draft tree hash, and
latest passing check hash. If any artifact changes after the decision, publish
verification must fail.

## Check Records

`argos provenance record-check` stores the full JSON output of
`argos knowledge check` under `checks/check-NNN.json`.

The check record must include enough information for future agents and PR
reviewers to understand:

- which design was checked
- which draft path was checked
- whether validation, policy, and findability passed
- what findings remained

Only `result: pass` is publishable in first release.

## Hashing

Argos must calculate hashes instead of asking agents to provide them.

Design hash:

```text
sha256(file bytes at subject.design_path)
```

Draft tree hash:

```text
sha256(canonical list of files under subject.draft_path + each file's bytes)
```

The canonical list should:

- use slash-separated relative paths
- sort paths bytewise
- include package entrypoints and supporting files
- reject paths outside the workspace

Check hash:

```text
sha256(file bytes of checks/check-NNN.json)
```

These hashes make publish fail if a design, draft, or check result is changed
after a decision is recorded.

## CLI Contract

Add a top-level `provenance` command group:

```bash
argos provenance start --design <design.json> --draft <draft-path>
argos provenance record-decision --provenance <id-or-path> \
  --stage <design|draft_write|publish> \
  --decision <approved|changes_requested|rejected> \
  --decided-by <actor> \
  --role <role> \
  --source <conversation|pull_request|issue|document|other> \
  --reason <text> \
  --recorded-by <actor>
argos provenance record-check --provenance <id-or-path>
argos provenance verify --provenance <id-or-path>
```

`start` creates `knowledge/.inbox/provenance/<provenance-id>/provenance.json`
and records the initial design hash. If the draft path does not exist yet, the
draft tree hash remains empty until the first check.

`record-decision` appends to `decisions.jsonl` and records hashes appropriate to
the stage:

- `design`: design hash
- `draft_write`: design hash
- `publish`: design hash, draft tree hash, latest check hash

`record-check` runs or reuses `argos knowledge check` through the same internal
logic, writes `checks/check-NNN.json`, updates `provenance.json`, and prints a
concise result.

`verify` recomputes hashes, checks required decisions, checks the latest check
result, and reports whether the provenance record is publishable.

## Publish Contract

The new first-release publish path is:

```bash
argos knowledge publish --provenance <provenance-id>
```

Publish must:

1. Load the provenance record from `knowledge/.inbox/provenance/<id>/`.
2. Verify the design hash.
3. Verify the draft tree hash.
4. Verify the latest check hash.
5. Require an approved `design` decision.
6. Require an approved `draft_write` decision.
7. Require the latest check result to be `pass`.
8. Require an approved `publish` decision bound to the current hashes.
9. Require the draft path to be under `knowledge/.inbox/items/` or
   `knowledge/.inbox/packages/`.
10. Move the draft to the official path.
11. Convert published knowledge to `status: active`.
12. Move the provenance record to
    `knowledge/provenance/<knowledge-id>/<provenance-id>/`.
13. Update `provenance.json` to `state: published`.
14. Revalidate the official target.
15. Roll back the draft and provenance move if activation or final validation
    fails.

`argos knowledge publish --design <design.json> --path <draft>` should no longer
be the primary public flow after provenance lands. Because the project has not
entered production, the implementation may either remove it or keep it as a
temporary compatibility path that clearly points agents to `--provenance`.

## Agent Flow

Personal mode:

```text
human states durable knowledge intent
agent designs knowledge
agent starts provenance
human approves design and draft write
agent writes inbox draft
agent records check
human approves publish
agent publishes with provenance
agent indexes and verifies findback
agent commits
```

Team PR mode:

```text
author and agent design knowledge on a branch
agent creates draft, provenance, check record, and publish decision record
agent publishes on the branch to create official candidate files
author opens PR
domain reviewer reviews knowledge diff and provenance diff
CI runs validate, provenance verify, index smoke, and relevant tests
merge to main makes the knowledge team accepted
```

Argos provenance prepares evidence for review. It does not make the authoring
agent the real reviewer.

## CI And PR Fit

First-release CI should be able to run:

```bash
argos validate
argos provenance verify --all
argos index
go test ./...
```

`--all` verifies both inbox provenance and published provenance records visible
in the checkout.

Repository controls remain external:

- PR review
- CODEOWNERS
- protected branches
- signed commits
- CI requirements

Argos should not depend on those systems in first release, but the provenance
files should make PR review easier and more structured.

## Query Behavior

Normal discovery should continue to query official knowledge only. Provenance is
not loaded into default `knowledge find` results.

Later query additions can expose provenance on demand:

```bash
argos knowledge provenance --json <knowledge-id>
```

First release does not need this command unless publish verification needs it.

## Migration From Design Review Fields

Current write design records include review-like fields such as design approval,
draft write approval, and publish approval. After provenance lands:

- Design files remain the place for intent, scope, sources, draft output, and
  check plan.
- Design files must not be treated as the authority for publish approval.
- Provenance decision records become the publish authority.
- Existing design fields may be accepted temporarily for old tests or old draft
  files, but new documentation and skills should direct agents to provenance.

The implementation should update README and `skills/capture-knowledge/SKILL.md`
so agents stop editing approval fields in design JSON as the source of truth.

## Testing

Required test coverage:

- `provenance start` creates a valid inbox provenance record.
- `record-decision` appends JSONL and records the correct stage hashes.
- `record-check` stores check JSON and updates latest check metadata.
- `verify` fails when design changes after decision.
- `verify` fails when draft content changes after check or publish decision.
- `verify` fails without design, draft_write, or publish approval decisions.
- `knowledge publish --provenance` moves draft and provenance together.
- publish rollback restores both draft and provenance on final validation
  failure.
- normal `knowledge find` does not load provenance as knowledge.
- team-mode fixture demonstrates official candidate files plus provenance in a
  branch-like workspace.

## Readiness Definition

This module is complete when:

- The official write path uses provenance as the publish gate.
- A fresh workspace can author, check, publish, index, and findback with
  provenance.
- The capture skill explains that Argos records provenance and PR review remains
  the team review mechanism.
- README explains Git as source of truth and SQLite as generated index.
- Existing design approval fields no longer serve as publish authority.
- Full tests pass.
